## 摘要

软件代码质量是一个包含三个维度的构念：可维护性（Maintainability）、可靠性（Reliability）和功能性（Functionality）。尽管许多企业已将代码质量指标纳入运营实践，但对这些指标的评估仍缺乏统一标准。我们将不同指标分为两类：1）单调指标（Monotonic Metrics）——对代码质量具有一致影响；2）非单调指标（Non-Monotonic Metrics）——与代码质量缺乏一致的对应关系。为了对二者进行统一评估，我们提出了一种基于分布的指标评分方法。实证分析涵盖 36,460 个高质量开源软件（OSS）仓库及其来自 SonarQube¹ 和 CK² 的原始指标数据。评估得分对软件采用情况具有良好的解释力。本研究丰富了代码质量的多维构念及其指标度量体系，为单调与非单调指标的一致性度量提供了实践指引。

**关键词** 开源软件、代码质量、构念测量、非单调指标

## 1 引言

代码质量是指代码编写良好、满足既定需求的程度 [1]。精确的代码质量度量可以改善软件产品、提高用户满意度并降低 IT 系统成本 [2]，进而影响软件的采用决策 [3, 4]。因此，众多企业已引入各种方法来评估代码质量，但这些方法呈现出高度多样性，缺乏一致标准。

图 1 表明，代码质量是一个多维构念，包含可维护性（Maintainability）、可靠性（Reliability）和功能性（Functionality）三个维度 [1]。基于代码质量维度度量的相关文献，我们识别出 20 个不同指标，并将其划分为单调指标和非单调指标。单调指标对代码质量有一致的影响，而非单调指标与代码质量之间缺乏一致的关系（图 2）。大多数单调指标与代码质量呈单调递减关系——以代码异味（Code Smells）数量为例，其增多通常意味着整体代码质量的下降。

现有文献在对两类代码质量指标进行一致评估的方法论上存在空白，尤其是针对非单调指标。因此，企业中评估代码质量最普遍的方法仍是同行代码审查 [5]。为了一致地评估两类指标，我们提出一种基于分布的方法，该方法在软件采用解释力方面表现优异。

**研究问题 1**
> 如何对软件代码质量中的单调指标和非单调指标进行一致评估？

我们通过分析高星数 OSS 中各指标的概率分布来评估指标得分。对于单调指标，拟合指数分布，以累积分布函数（CDF）中距阈值参数的加权距离作为得分；对于非单调指标，拟合非对称高斯分布，以 CDF 中距中心点的加权距离作为得分。每个指标的评估得分范围为 0 ∼ 100。

我们在 36,460 个 GitHub OSS 仓库上进行了实证分析。选取高星数仓库使评估更为严格，因为这些高质量仓库作为参考基准，质量略逊于所选仓库的项目通常会因我们较陡峭的分布而获得极低的得分。

**研究问题 2**
> 评估得分对软件采用的影响是什么？

我们研究了代码质量指标得分对 OSS 星数的解释力。星数反映了 OSS 的质量和采用程度 [6]。采用标准机器学习方法，使用 R²（决定系数）和准确率评估解释能力。结果表明，我们的代码质量得分能够很好地解释 OSS 的星数。该方法可应用于不同的目标变量，在各类场景中提供灵活的评估策略。

本研究的主要贡献如下：已有文献讨论了多种代码质量指标 [7, 8, 9]，但缺乏一致的指标评估方法。我们将代码质量指标划分为两类，并用新颖的基于分布的方法对其进行评估；在 36,460 个 GitHub OSS 仓库上开展了实证分析；通过评估得分解释了 OSS 采用情况 [1]，揭示了代码质量对 OSS 采用的潜在影响。本研究深化了对两类代码质量指标的理解，并有助于建立更好的质量控制标准与实践。

## 2 文献综述

开源软件（OSS）采用社区化的软件开发模式，这显著提升了其代码质量 [10, 11]，并推动了创新 [12, 13, 14]。许多企业和开发者积极贡献并使用 OSS [15]，从而持续提升 OSS 整体代码质量。OSS 被广泛复用，因为许多 OSS 具有高代码质量 [16]，能有效降低开发者的搜索成本 [17, 18]。因此，我们以 OSS 作为评估软件代码质量的基准。

已有大量研究关注软件的性能评估 [19]。不恰当的性能度量被认为是 IT 系统失败的主要原因之一 [2, 20]。随着区块链系统 [21]、AI 智能体系统 [22] 等新技术的出现，对软件代码质量进行更精确度量的需求日益迫切 [23]。本研究与以往研究的不同之处在于：我们考虑了高质量软件的分布特征，并为每个软件提供了精确评分。

代码质量涵盖多个维度 [24]。IEEE 标准将代码质量定义为满足既定需求的软件特征和属性的集合 [25]。此后，用户友好性和实用功能也被纳入代码质量定义 [1]，与 ISO/IEC 25010 标准 [26] 中的三个维度相呼应：可维护性（Maintainability）、可靠性（Reliability）和功能性（Functionality）[27]。其他研究也提出了类似的维度划分：可维护性 [28]、可读性（Readability）[29] 和功能性 [30]。我们基于上述文献，在表 1 中定义了构念及其维度。

**表 1：构念定义**

| 构念 | 定义 | 维度 | 定义 |
|------|------|------|------|
| 代码质量（Code Quality） | 代码编写良好、满足既定需求的程度 [1] | 可维护性（Maintainability） | 代码易于理解、增强或修正 [31] |
| | | 可靠性（Reliability） | 代码对用户友好且稳定 [1] |
| | | 功能性（Functionality） | 代码具有实用功能 [1] |

代码质量的指标包括组件大小 [8]、代码复杂度 [7, 9] 等。然而，现有的大多数指标识别研究侧重于单调指标，而非非单调指标，因为单调指标与软件代码质量具有一致的关系。本文同时考虑两类指标，并提出了统一的评估方案。

反映性度量指标（如星数 [6]）可以反映软件代码质量的整体水平。尽管 OSS 的采用行为受到多种因素影响，如贡献度 [32]、透明度 [33] 和领导者资源 [34]，但 OSS 采用决策能够体现良好的 OSS 代码质量。Lee 等 [1] 强调了代码质量对用户满意度和采用决策的影响。采用率最高的 OSS 往往是代码质量卓越的项目。因此，我们使用 GitHub 星数作为代码质量的反映性度量指标。

## 3 研究方法

本研究以星数作为识别高质量仓库的反映性度量指标，将代码质量指标划分为两类，分析各类指标的分布特征，并提出统一的基于分布的评估方法。

我们首先绘制每个指标在高星数 OSS 仓库中的分布，然后根据其对应指标的 CDF 对各仓库进行评分。

表 3 呈现了两类指标分布：单调指标与非单调指标。对单调指标拟合指数分布，其概率密度函数（PDF）为：

$$f_1(x; c, \lambda) = \begin{cases} 0 & \text{若 } x \leq c \\ \lambda \exp[-\lambda(x - c)] & \text{若 } x > c \end{cases} \tag{1}$$

其中 λ 和 c 为拟合参数。基于公式（1）的 CDF，对应的评分函数为：

$$M_1(x; c, \lambda) = 100 \times \begin{cases} 1 & \text{若 } x \leq c \\ \exp[-\lambda(x - c)] & \text{若 } x > c \end{cases} \tag{2}$$

得分范围为 0 ∼ 100，在 c 处取峰值，并在 x > c 时呈指数衰减。

> **图 2：非单调指标分布与单调指标分布示例**
>
> （原文图 2 包含两组密度分布直方图，左侧为非单调指标——平均圈复杂度（Average Cyclomatic Complexity），右侧为单调指标——每千行重复行数（Duplicated Lines per 1000 Lines），均按 Java、JavaScript、Python、TypeScript 四种语言分组展示。MD 无法渲染原图，请参阅原文 PDF 第 4 页 Figure 2。）

非单调指标遵循非对称高斯分布（见图 2 左侧），其 PDF 为：

$$f_2(x; \mu, \sigma_1, \sigma_2) = \begin{cases} \dfrac{1}{\sqrt{2\pi}} \cdot \dfrac{2}{\sigma_1+\sigma_2} \exp\!\left(-\dfrac{(x-\mu)^2}{2\sigma_1^2}\right) & \text{若 } 0 \leq x < \mu \\ \dfrac{1}{\sqrt{2\pi}} \cdot \dfrac{2}{\sigma_1+\sigma_2} \exp\!\left(-\dfrac{(x-\mu)^2}{2\sigma_2^2}\right) & \text{若 } x \geq \mu \end{cases} \tag{3}$$

其中 μ、σ₁、σ₂ 为拟合参数，分别表示峰值位置、右侧峰高以及两侧的峰宽。对应的评分函数为：

$$M_2(x, \mu, \sigma_1, \sigma_2) = 100 \times \begin{cases} 1 - \text{erf}\!\left(\dfrac{x-\mu}{\sigma_1\sqrt{2}}\right) & \text{若 } 0 \leq x < \mu \\ 1 - \text{erf}\!\left(\dfrac{x-\mu}{\sigma_2\sqrt{2}}\right) & \text{若 } x \geq \mu \end{cases} \tag{4}$$

得分范围为 0 ∼ 100，在 μ 处取峰值，并按高斯函数两侧的 Z 分数衰减。

为获得综合得分，我们对各单项得分赋予权重。仓库 k 的综合得分计算如下：

$$Q_k^{overall} = \sum_i \omega_i \cdot Q_{i,k}^{metric}, \quad \text{约束条件：} \sum_i \omega_i = 1 \tag{5}$$

权重 ωᵢ 由监督学习模型针对目标变量（如仓库星数）的指标得分重要性值导出。

## 4 实证分析

### 4.1 数据来源

GitHub 是全球最大的 OSS 管理平台，拥有超过 3900 万个公开仓库（截至 2023 年 6 月）。我们选取以 Java、Python、JavaScript 和 TypeScript 为主要编程语言的仓库子集，并按星数排序，为每种语言收集了排名前约 20,000 的仓库代码。GitHub 星数是衡量 OSS 采用程度的指标 [6]。我们通过模式匹配去除了非工程类仓库，例如 JavaGuide 中的 Java 面试指南。

我们使用代码扫描工具获取指标。脚本语言仓库（Python、JavaScript、TypeScript）可直接导入，而非脚本语言 Java 仓库需先编译。由于不同仓库的 JDK、Maven 或 Gradle 版本各异，编译 Java 仓库颇具挑战。因此，我们仅选取有 GitHub Release 的仓库进行编译，最终得到 36,460 个仓库，代码总行数超过 6 亿行。表 2 列出了克隆仓库的统计摘要。仓库星数最低值均超过 50，与整体 OSS 仓库相比，代码质量良好。

**表 2：统计摘要**

| 编程语言 | 最高星数 | 最低星数 | 筛选后仓库数 |
|----------|---------|---------|------------|
| Java | 50k | 100 | 1,645 |
| Python | 228k | 260 | 16,096 |
| JavaScript | 107k | 270 | 7,722 |

### 4.2 指标概览

我们使用 SonarQube 和 CK 从 OSS 仓库中提取指标。对于 Java 仓库，共生成超过 100 个指标，并依据 ISO/IEC 25010 国际标准 [26] 选取了 20 个。对于脚本语言仓库，仅提取了 12 个指标。表 3 列出了 20 个指标及其对应的 ISO/IEC 25010 特性。

为确保评分公平，我们对指标进行了归一化处理：圈复杂度（Cyclomatic Complexity）、认知复杂度（Cognitive Complexity）、代码异味（Code Smells）、待覆盖行数（Line to Cover）和违规相关指标按非注释代码行数归一化；重复行数（Duplicated Lines）按代码总行数归一化；注释行数（Comment Lines）按非注释行数与注释行数之和归一化（以消除仓库规模差异）。文件复杂度（File Complexity）和重复文件数（Duplicated Files）按文件数归一化，重复代码块（Duplicated Blocks）按语句数归一化（以消除仓库间差异）。此归一化过程使不同 OSS 的指标得分更具无偏性。

### 4.3 重要性权重

我们使用标准机器学习方法为不同指标得分导出权重，并计算仓库的整体代码质量得分。模型以评估得分解释 OSS 采用情况（GitHub 星数）。

> **图 3：以 GitHub 星数为目标变量的代码质量评分工作流**
>
> （原文图 3 为流程图，展示从 GitHub 仓库出发，经过过滤→训练/测试数据、新数据→指标扫描器→指标提取，分三条路径（可维护性 Maintainability、可靠性 Reliability、功能性 Functionality）各自评分，生成原始指标数据→指标得分，再经分类（GitHub Star）→指标重要性加权→最终代码质量得分，并有更新反馈。MD 无法渲染原图，请参阅原文 PDF 第 5 页 Figure 3。）

图 3 展示了从数据采集到最终得分的全过程。我们使用自定义数据过滤器确保保留真正的工程类仓库；使用指标扫描器提取代码质量指标，并根据第 3 节的基于分布方法生成指标得分（每种编程语言的每个指标均有其独立分布）。我们以 GitHub 星数为依据，实现基于 0-1 标签的梯度提升分类器（GBC，Gradient Boosting Classifier）模型——将星数排名前 20% 和后 20% 的仓库分别标记为 1 和 0。模型输出各指标的重要性值作为权重，最终依据公式（5）计算加权平均代码质量得分。

GBC 算法见算法 1，其中每个数据点包含指标得分 **m**ᵢ 及其对应的 GitHub 星数分类 cᵢ。训练集（D）与验证集比例为 4:1。GBC 算法以集成模型 F₀(**m**) 为基础，通过将基学习器 hₜ(**m**) 拟合到损失函数的负梯度来进行微调。学习率 η 决定基学习器的贡献程度，最终集成模型 F(**m**) 提供汇总预测。

**算法 1：本文中的 GBC 算法**

```
输入：训练数据集 D = {(mᵢ, cᵢ)}ᴺᵢ₌₁，迭代次数 T
输出：集成模型 F(m)

初始化模型 F₀(m) = 0
for t = 1 to T do
    计算负梯度：rᵢₜ = -∂L(cᵢ, F(mᵢ))/∂F(mᵢ)|F(m)=Fₜ₋₁(m)
    将基学习器 hₜ(m) 拟合到负梯度：hₜ(m) = argminₕ Σᴺᵢ₌₁ L(cᵢ, Fₜ₋₁(mᵢ) + h(mᵢ))
    更新集成模型：Fₜ(m) = Fₜ₋₁(m) + η·hₜ(m)，其中 η 为学习率
end
```

## 5 结果

### 5.1 指标分布

我们在 36,460 个 GitHub OSS 仓库上进行了实证分析。高星数仓库的选取使评估更为严格，因为这些仓库整体性能更优，导致分布更为陡峭。

表 4 和表 5 分别列出了非对称高斯分布（公式 3）和指数分布（公式 1）的拟合参数。Java 仓库额外包含 8 个描述代码耦合与内聚的可维护性指标，其他编程语言因缺乏合适的指标扫描器而未涵盖。

单调指标（如"代码异味（Code Smells）"）呈现指数分布特征（见图 2 和表 5），与"代码质量越高、缺陷越少"的认知一致，验证了本方法的有效性。此外，阈值参数 c 反映了获得满分的容忍值——在公式（1）的概率密度函数中，除"代码异味（Code Smells）"、"继承树深度（Depth Inheritance Tree）"和"总违规数（Total Violations）"外，c 近似为 1。

拟合的指数衰减参数 λ 反映了指标对得分的敏感程度。对于"文件复杂度（File Complexity）"、"继承树深度（Depth Inheritance Tree）"、"子类数量（Number of Children）"、"重复代码块（Duplicated Blocks）"和"重复文件（Duplicated Files）"等指标，λ ≲ 1，意味着对量级为 1 的指标变化敏感度较低；而"总违规数（Total Violations）"的 λ 值较高，反映出违规数量的高敏感性。

非单调指标（如"圈复杂度（Cyclomatic Complexity）"）遵循非对称高斯分布。根据公式（4），指标值接近高斯中心的仓库得分更高，因为它们落在高质量 OSS 集中分布的区间内。从表 4 可以看出，对于大多数编程语言，"圈复杂度（Cyclomatic Complexity）"、"认知复杂度（Cognitive Complexity）"和"注释行数（Comment Lines）"的高斯中心 μ 较大（≫ 1），但 JavaScript 和 TypeScript 的"注释行数（Comment Lines）"例外（μ ≈ 0），这可能是因为这两种语言通常较易理解，不需要额外的注释行。

拟合宽度 σ₁,₂ 较大且具有非对称敏感性，即非对称高斯分布右侧的尾部较长。以 Python 的"注释行数（Comment Lines）"为例，在中心点左侧增加注释行数时敏感度较高，而在中心点右侧则敏感度降低。

### 5.2 重要性权重

表 6 列出了第 4 节中 GBC 模型的特征重要性值，用作公式（5）中三个维度（可维护性、可靠性、功能性）内的指标得分权重。我们对每个维度内的重要性值进行了归一化处理，以获得维度内的相对权重。

在可维护性（Maintainability）维度，"文件复杂度（File Complexity）"在四种编程语言中均具有最大权重，其次依次为"认知复杂度（Cognitive Complexity）"、"圈复杂度（Cyclomatic Complexity）"和"代码异味（Code Smells）"，这些指标对可维护性得分贡献更大。对于 Java 仓库，所有耦合与内聚指标的贡献均 ≲ 0.1，反映了其对 OSS 采用的贡献较弱。

在可靠性（Reliability）维度，"总违规数（Total Violations）"对 Java 贡献最大，而"严重违规数（Critical Violations）"对其他三种语言贡献最大，这表明不同语言在解决违规问题上的优先级存在差异。

在功能性（Functionality）维度，"注释行数（Comment Lines）"对 Java 贡献更大，这可能是因为 Java 的直觉性较差，需要代码注释来辅助理解。"注释行数（Comment Lines）"对其他三种脚本语言也有显著贡献。我们注意到，原始数据中"待覆盖行数（Line to Cover）"指标值均为零，可能是由于获取该指标时存在问题，或者是因为 OSS 仓库中的代码很少经过测试。当将本方法应用于企业内部的闭源仓库时，这一缺口可以得到弥补。

### 5.3 软件采用

我们在图 4 中展示了所有 OSS 仓库的综合得分分布，并通过表 7 评估指标得分对 OSS 星数的解释力。我们发现，Java 代码指标得分对 OSS 仓库星数的解释力高于其他语言，这表明代码质量能够更好地决定 Java 类 OSS 在星数方面的成功，这可能源于 Java 可用指标更多，或者 Java 仓库通常用于构建大型平台和系统的特性。

相比之下，JavaScript、Python 和 TypeScript 的指标得分解释力相对较低，表明代码质量对这些语言的 OSS 采用决定性可能较弱，这或许与它们主要用于数据分析等领域有关——在这些领域中，采用决策受代码质量影响较小。

## 6 结论

本研究聚焦于代码质量的三个维度：可维护性（Maintainability）、可靠性（Reliability）和功能性（Functionality）。我们基于指标分布对各指标进行评估。研究深化了对代码质量的理解，有助于建立更好的质量控制标准与实践，最终支持软件的成功与可持续发展。

尽管本研究提供了有价值的启示，但也存在一些局限性需要承认。我们尚未系统验证该方法的有效性。未来研究可引入敏感性测试等验证技术，以确保分布拟合的准确性和可靠性。此外，拟合分布的参数对数据分布较为敏感，需要纳入更多数据来确定参数。
